### 2. 🟢 Node.js — What It Is and Why We Use It

The transition into the new topic started with the question: you know JavaScript — but you learned it *in the browser*. What if you want to run JavaScript *without* a browser?

**Node.js** is a runtime environment that lets you run JavaScript anywhere — the terminal, a server, a script — without a browser. This is what makes it possible to write backend web servers in JavaScript.

![Node.js logo](https://i.imgur.com/bAbXv7e.png)

**Installing Node.js:**
```bash
# Install via winget
winget install OpenJS.NodeJS

# Verify the install worked (open a new terminal tab first if fresh install)
node -v     # prints Node.js version
npm -v      # prints npm (Node Package Manager) version
```

**LTS vs Latest — which version to use?**

| Version type | What it is | When to use |
|---|---|---|
| **LTS** (Long Term Support) | Locked in, receives only security + bug fixes for years | Production software, client projects — anything that needs to stay stable |
| **Latest** | Newest features, may change rapidly | Learning, school projects — you want the newest features and long-term you'll update |

For school: use **Latest**. By the time you graduate, those features will be in the next LTS anyway. The key warning: if you install the latest version and something breaks overnight — check if a new *major version* just dropped. That happened to a student in a previous class (installed in the evening, broke the next morning). If a fresh `x.0.0` just released, wait a couple of months for the initial bugs to be patched.

**Using VS Code's Git GUI for commits:**
The session used this as an opportunity to show the VS Code Source Control panel as an alternative to the command line.

- The "Stage Changes" `+` button next to a file = `git add <file>`
- "Stage All Changes" = `git add .`
- The message box + "Commit" button = `git commit -m "message"`
- "Sync Changes" = `git push` (and pull, since it syncs both directions)
- If you press Commit without a message, VS Code opens a file editor — write the subject on line 1, blank line, then body; `Ctrl+S` to save, close the tab to complete

**Why learn the command line if buttons exist?**
The GUI handles simple, everyday operations well — the instructor uses it too. But for complex operations (rebase, conflict resolution, interactive history editing, `git reflog`) the GUI doesn't expose the tools at all. When something goes seriously wrong, only the command line can fix it.

> ⚠️ **Nested repo warning:** Don't create a Git repo inside another Git repo's folder. VS Code will get confused and you'll get errors. Always work in dedicated project folders.

### 4. 🖥️ Writing Your First Node.js Web Server

The first server code was taken directly from the Node.js homepage example. Create a file called `server.mjs`:

```js
import { createServer } from 'node:http'

const hostname = '127.0.0.1'
const port = 3000

const server = createServer((req, res) => {
  res.statusCode = 200
  res.setHeader('Content-Type', 'text/plain')
  res.end('Hello World\n')
})

server.listen(port, hostname, () => {
  console.log(`Server running at http://${hostname}:${port}/`)
})
```

**Why `.mjs`?**
There are two module systems in JavaScript: **ES Modules** (modern, uses `import`/`export`, `.mjs` extension) and **CommonJS** (older, uses `require()`, `.cjs` extension). `.mjs` tells Node.js to treat the file as an ES Module — the same system Vue uses. You can also use `.js`, but the extension makes the intent explicit.

**Running the server:**
```bash
node server.mjs
# Output: Server running at http://127.0.0.1:3000/
```
Open `http://127.0.0.1:3000` (or `http://localhost:3000`) in the browser — you'll see "Hello World". To stop the server: `Ctrl+C`.

**Port conflict:**
If you try to run two copies of the server at the same time on port 3000, the second one throws `EADDRINUSE: address already in use`. Only one process can listen on a given port at a time.

### 5. 🌐 Networking Concepts — IP Addresses, DNS, Ports

The rest of the session was a practical networking explainer to understand how servers actually communicate.

#### IP Addresses

Every device on a network has an **IP address**. Two types matter:

**Local (private) IP** — used inside your network:
- `127.0.0.1` — the loopback / localhost address. This means "this computer itself". Present on every machine on every OS. `localhost` is a hostname alias for it — they're interchangeable.
- `192.168.1.x` — typical home network range assigned by your router
- `172.18.x.x` — common school/enterprise network range

**Public IP** — what the internet sees:
- Visit "what's my IP" to see it
- Different at school, home, and on mobile data
- This is what external connections reach you through

You can discover your machine's local IP addresses by running:
```bash
ipconfig          # Windows / Bash
```

#### DNS — Domain Name System

Humans use names like `google.com`; computers need IP addresses like `142.251.38.78`. **DNS** is the phone book that translates between them.

When you type a URL:
1. Your computer checks its **hosts file** first (a local file — this is also where `localhost` → `127.0.0.1` is defined)
2. If not found there, it asks its **DNS server** (usually your router, or Google's `8.8.8.8`)
3. That DNS server may ask other DNS servers up the chain
4. Eventually the IP address comes back — instantly, from your perspective

You can verify DNS resolution yourself:
```bash
ping google.com     # shows the IP address it resolves to
```

#### Port Numbers

An IP address gets you to a *machine*. A **port number** tells that machine which service you want to talk to. One machine can run many servers at once, each on a different port.

Port numbers range from 0 to 65535. Convention:
- **Under 1000** — reserved/well-known services. Don't use these for your own servers.
- **1000–49151** — registered ports. Many common tools live here.
- **49152–65535** — dynamic/private range. Use these freely.

Key port numbers to know:

| Port | Protocol | Notes |
|------|----------|-------|
| 21 | FTP | File Transfer Protocol |
| 22 | SSH | Secure Shell — remote terminal access |
| 25 | SMTP | Email sending |
| 80 | **HTTP** | Standard web — browsers try this by default |
| 443 | **HTTPS** | Secure web — browsers prefer this today |
| 3000 | (dev) | Common for Node.js/React dev servers |
| 8000 / 8080 | (dev) | Common dev server alternatives |
| 25565 | Minecraft | The default Minecraft multiplayer port |

> 💡 Why is the lower the number the older/more important? Because engineers numbered things in order — the first protocol got port 1, the next got port 2, etc. The older and more fundamental a service, the lower its port.

When you type a URL in the browser without a port number, it automatically tries **443** first (HTTPS), then **80** (HTTP). If your server is on port 3000, you have to tell the browser: `localhost:3000`.

#### TCP — Transmission Control Protocol

Underlying almost all of this is **TCP**. It's the base-level protocol that:
1. Establishes a connection between two machines
2. Sends data back and forth as a stream of packets
3. Guarantees delivery and correct ordering
4. Closes the connection

HTTP, HTTPS, SSH, FTP — they all run on top of TCP. TCP is very old and very reliable — it's what the internet is built on.

### 6. 📡 HTTP — How Web Requests Actually Work

With TCP as the foundation, **HTTP** defines the *language* of web communication: what a request looks like and what a response looks like.

**A typical HTTP request (browser → server):**
```
GET / HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html, application/xhtml+xml, ...
Accept-Language: et, en-US;q=0.9
Accept-Encoding: gzip, deflate, br
Connection: keep-alive
```

Fields explained:
- `GET /` — the HTTP method (`GET`, `POST`, `PUT`, `DELETE`, etc.) and the path being requested
- `Host` — the domain name requested
- `User-Agent` — which browser/client is making the request
- `Accept` — what content types the browser is willing to receive (HTML preferred, falls back to XML, etc.)
- `Accept-Language` — the browser tells the server what language the user prefers. This is how websites auto-show your language.
- `Accept-Encoding` — the browser says it can handle compressed responses (gzip etc.)

**A typical HTTP response (server → browser):**
```
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 1234
Server: Apache/2.4
Last-Modified: Mon, 28 Apr 2026 10:00:00 GMT

<html>...
```
- Status code `200 OK` — success. Other important codes: `404 Not Found`, `500 Internal Server Error`, `301 Moved Permanently`, `403 Forbidden`
- `Content-Type` — tells the browser what it's receiving
- `Content-Length` — how many bytes to expect
- Then a blank line, then the actual content (HTML, JSON, etc.)

**You can see this in action:** open browser DevTools → Network tab → refresh the page → click any request → see Headers. Request headers show what the browser sent; response headers show what the server replied.

**Why compression matters:**
Computers and phones are very fast — far more powerful than the computers that sent humans to the moon. What's slow is the *network*. So servers compress responses (zip them up) before sending, and browsers unzip on arrival. The CPU cost of compression is negligible; the bandwidth saved is significant.
