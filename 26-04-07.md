### 2. ⚡ Git Aliases & `.zshrc`

Before diving into conflicts, the instructor introduced **Git aliases** — custom shortcuts for long Git commands — and showed how to add them to `.zshrc` so they persist across terminal sessions.

**Why aliases:**
- Commands like `git log --oneline --all --graph` are long and tedious to type every time
- Aliases let you define a short version, e.g. `gl` → runs the full log command
- The instructor noted this is part of the bigger programmer habit of automating repetitive work

**How to set up:**
```bash
# Open .zshrc in editor
code ~/.zshrc

# Add aliases at the bottom
alias gl="git log --oneline --all --graph"
alias gs="git status"
```

After saving, run `source ~/.zshrc` to apply the changes immediately without reopening the terminal. From then on, typing `gl` runs the full log command.

> 💡 The instructor's point: the best programmers always find ways to eliminate repetitive typing — aliases are a small but meaningful example of that mindset.

### 3. 💥 Merge Conflicts — What They Are

The core topic of the session was **merge conflicts** — what they are, why they happen, and how to resolve them.

**When do conflicts happen:**
- When two branches have both changed the **same lines** of the same file
- Git can merge branches automatically most of the time, but when the same section has been edited differently on both sides, Git can't decide which version is correct — it stops and asks you to decide

**What a conflict looks like in a file:**
When a conflict occurs, Git inserts conflict markers directly into the file:
```
<<<<<<< HEAD
your current branch's version of the line
=======
the incoming branch's version of the line
>>>>>>> feature-branch
```

- Everything between `<<<<<<< HEAD` and `=======` is your current branch's content
- Everything between `=======` and `>>>>>>>` is the incoming branch's content
- Your job is to delete the markers and decide what the final version should look like

**Resolving a conflict manually:**
1. Open the file — find the conflict markers
2. Decide which version to keep (or combine both)
3. Delete all the marker lines (`<<<<`, `====`, `>>>>`)
4. Save the file
5. `git add <file>` — mark the conflict as resolved
6. `git commit` — complete the merge

**Resolving in VS Code:**
VS Code highlights conflict markers with colour and provides clickable options:
- **Accept Current Change** — keep only your branch's version
- **Accept Incoming Change** — keep only the other branch's version
- **Accept Both Changes** — keep both versions one after the other
- **Compare Changes** — side-by-side diff view for complex conflicts

The instructor demonstrated all four options live, noting that "Accept Both" is occasionally useful but often leaves messy duplicated content that still needs manual tidying.

### 4. 🔀 `git rebase` — A Cleaner Alternative to Merge

After covering merge conflicts, the instructor introduced **`git rebase`** — a different way to incorporate changes from one branch into another.

**The key difference between merge and rebase:**

| `git merge` | `git rebase` |
|-------------|--------------|
| Creates a new merge commit | Rewrites commits so they appear after the target |
| History shows branching | History looks linear — as if you worked on main all along |
| Safe for shared branches | Should **only** be used on your own local branches |

![Merge vs Rebase diagram](https://opensource.com/sites/default/files/uploads/merge-commit-vs-rebase.png)

**How rebase works conceptually:**
- Instead of creating a new commit that combines both branches, rebase replays your commits one by one on top of the target branch
- The result looks like your feature branch was always built from the latest version of main
- This makes `git log` much cleaner — no merge commits, no branching lines

**Basic usage:**
```bash
# While on your feature branch:
git rebase main
# This replays your branch's commits on top of the latest main
```

**When rebase also creates conflicts:**
- Rebase can still hit conflicts — the same file, same lines, different content
- But instead of one big merge conflict, you resolve conflicts commit by commit
- After resolving each conflict: `git add <file>` → `git rebase --continue`
- The result after a successful rebase: a clean, linear commit history, no conflicts

**When not to rebase:**
- Never rebase a branch that others are also working on — it rewrites history and causes serious confusion for anyone who has pulled the old commits

### 5. ✏️ `git commit --amend` — Fixing the Last Commit

Near the end of the session, the instructor covered **`git commit --amend`** — a way to fix the most recently made commit.

**What it can do:**
- Fix a typo in a commit message
- Add a forgotten file to the last commit without creating a new commit
- Combine "oops" fixes into the commit they belong to

**Usage:**
```bash
# Fix only the commit message:
git commit --amend -m "Corrected commit message"

# Add a forgotten file to the last commit:
git add forgotten-file.txt
git commit --amend --no-edit   # keeps the original message
```

**Important caveat:**
`--amend` rewrites the last commit — it changes its hash. If you've already pushed that commit to GitHub, you'd need to force-push (`git push --force`), which can cause problems for others. The instructor's advice: use `--amend` only on commits that haven't been pushed yet.

**`git commit --amend` with no `-m` flag:**
Opens the editor (VS Code if configured) with the current commit message pre-filled, letting you edit it and save — similar to a normal commit but replacing the last one instead of creating a new one.

### 6. 🗜️ `git rebase -i` — Interactive Rebase (squashing commits)

The session ended with a brief introduction to **interactive rebase** (`git rebase -i`), a powerful way to clean up commit history before pushing.

**What it lets you do:**
- **Squash** multiple small commits into one clean commit
- **Reword** commit messages
- **Reorder** commits
- **Drop** commits entirely

**Usage:**
```bash
# Interactively rebase the last N commits:
git rebase -i HEAD~3   # opens editor showing last 3 commits
```

In the editor, each commit is listed with the word `pick` in front. You can change `pick` to:
- `squash` (or `s`) — combine this commit with the one above it
- `reword` (or `r`) — keep the commit but change its message
- `drop` (or `d`) — delete the commit entirely

The instructor demonstrated squashing several small "line 1", "line 2" type commits into a single clean commit — showing how much tidier `git log` looks afterwards. The use case: you've been committing frequently while working but want to push a single meaningful commit to share with others.
