### 2. 🏷️ Git Tags

The main new Git topic was **tags** — a way to mark a specific commit permanently with a meaningful name, most commonly used for versioning.

**Why tags exist:**
A tag always points to the same commit — unlike a branch, which moves forward with every new commit. This makes tags ideal for marking releases: version 1.0.0, version 2.1.3, etc. When someone downloads version 1.0.0 of your software a year from now, they get exactly the code that was tagged at that moment, no matter how many commits have been added since.

**Two types of tags:**

| Type | Command | Description |
|------|---------|-------------|
| Lightweight | `git tag v1.0.0` | Just a pointer to a commit — no extra metadata |
| Annotated | `git tag -a v1.0.0 -m "Release 1.0.0"` | Includes tagger name, date, and message — preferred for releases |

**Other useful tag commands:**
```bash
git tag                          # list all tags
git tag v1.0.0                   # create lightweight tag at HEAD
git tag -a v1.0.0 -m "message"   # create annotated tag
git push origin v1.0.0           # push a specific tag to GitHub
git push origin --tags           # push all tags at once
git checkout v1.0.0              # check out the state of the repo at that tag
```

**Tags are not pushed automatically.** A regular `git push` does not send tags — you must push them explicitly with `git push origin <tagname>` or `git push origin --tags`.

### 3. 🔢 Semantic Versioning (SemVer)

Linked closely to tags, the instructor introduced **Semantic Versioning** — the widely adopted standard for version numbers in software.

A SemVer version looks like: **`MAJOR.MINOR.PATCH`** — for example `2.4.1`

| Number | When to increment | Example scenario |
|--------|-------------------|-----------------|
| **MAJOR** | Breaking change — existing code using your library may stop working | Renamed a function, changed how an API works |
| **MINOR** | New functionality added in a backward-compatible way | Added a new function that doesn't break anything existing |
| **PATCH** | Backward-compatible bug fixes only | Fixed a bug, no new features |

**Key rules:**
- Once a version is released (tagged and pushed), never modify it — create a new version
- `0.x.y` versions are considered unstable/development — anything can change
- `1.0.0` is the first stable public release

The instructor noted that while not every personal project needs strict SemVer, it's nearly universal in professional and open-source software. Understanding it is essential for reading package changelogs and `package.json` version strings.

### 4. ✍️ How to Write Good Commit Messages

A dedicated section was spent on **commit message conventions** — one of the most visible professional habits a developer can build.

**The rules covered:**

**1. Write in the imperative (käskiv kõneviis)**
The subject line should complete the sentence: *"If applied, this commit will..."*
- ✅ `Add pagination component`
- ✅ `Fix null pointer when search is empty`
- ❌ `Added pagination` (past tense)
- ❌ `Adding pagination` (present continuous)

This convention comes directly from how Git itself writes its own messages (`Merge branch`, `Revert "..."`) and is now the de facto standard.

**2. Don't end the subject line with a period**
It's a title, not a sentence.

**3. Keep the subject line under 72 characters**
Many tools (GitHub, terminal `git log`) wrap or truncate at ~72 chars. VS Code extensions can show a ruler at this length to help.

**4. One logical change per commit**
Each commit should represent one focused change. Don't mix "add feature X" and "fix bug in Y" in the same commit — they belong in separate commits.

**5. Use the body for the *why*, not the *what***
The diff already shows *what* changed. The commit body (if needed) should explain *why* the change was made, or any non-obvious context.

The instructor showed how this applies to real commit history — a well-written log reads almost like a project diary, while a log of "asdf", "changes", "fix", "more stuff" tells you nothing.

### 5. 🕰️ `git checkout <hash>` — Visiting Old Commits

The session also covered how to **navigate back to any point in history** using `git checkout` with a commit hash.

```bash
git log --oneline        # find the commit hash you want
git checkout abc1234     # travel to that commit
```

This puts Git into **detached HEAD** state — you're no longer on a branch, you're floating at a specific commit. You can look around, run the code, read files — but you're not "on" a branch.

**What detached HEAD means:**
- Any new commits you make here won't be attached to any branch — they'll appear to "disappear" when you switch back
- The commits aren't actually deleted — they linger in Git's reflog for a while — but without a branch or tag pointing to them they become inaccessible through normal commands
- To go back: `git checkout main` (or `git switch main`)

**If you accidentally lose commits this way:**
- `git reflog` lists everything HEAD has pointed to recently — you can find the hash and use `git checkout <hash>` or create a new branch from it to rescue the work

### 6. ↩️ `git reset --hard` — Discarding Changes

The instructor also demonstrated **`git reset --hard`** — the command for throwing away commits or uncommitted changes completely.

```bash
git reset --hard HEAD        # discard all uncommitted changes (go back to last commit)
git reset --hard HEAD~1      # throw away the last commit AND its changes
git reset --hard <hash>      # go back to any specific commit, discarding everything after it
```

**The key distinction — soft vs hard:**

| Flag | What it does |
|------|-------------|
| `--soft` | Moves the branch pointer back but keeps the changes staged |
| `--mixed` (default) | Moves the branch pointer back, unstages changes, keeps files |
| `--hard` | Moves the branch pointer back AND deletes the changes from disk |

**When not to use `--hard` on a pushed branch:**
If the commits you're resetting away have already been pushed, this rewrites history on the remote too (via a force push) — which causes problems for anyone else who has pulled those commits. Use it freely on your own local work; be very careful with shared branches.
