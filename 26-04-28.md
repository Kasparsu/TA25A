## 🗂️ Topics Covered

### 1. 🖥️ Virtual Machines — The Old Way

The session opened with students who had already played with virtual machines (VirtualBox) as homework. VMs were used to frame the problem that containers solve.

**What a virtual machine does:**
A VM is a full computer inside your computer. It has its own **hypervisor** — specialised software that sits between your hardware and the VM guest OS — and runs a completely separate operating system on top of it. VirtualBox on your laptop is a type-2 hypervisor (it runs on top of Windows). Dedicated server hypervisors (type-1) have no host OS underneath — the hypervisor *is* the OS, and it runs directly on the hardware.

**The problem with VMs:**
A full virtual machine installs an entire operating system. That means:
- **Enormous disk usage** — a typical Linux VM takes 10–30 GB minimum once you add software, databases, etc.
- **Slow setup** — installing and configuring takes 10–15 minutes even with automation tools like Vagrant
- **Heavy resource use** — a VM consumes significant RAM and CPU for its own OS overhead

For a developer who wants to run a database or a web server for testing, spinning up an entire OS for each service is excessive.

**Vagrant** — a tool built on top of VirtualBox that automates VM setup via a configuration file (`Vagrantfile`). Instead of clicking through an installer, you describe the machine in code and Vagrant provisions it automatically. Still takes time, still takes disk space — it just removes the manual clicking.

### 2. 📦 Containers — A Lighter Alternative

The solution to the VM overhead problem is **containers**. The instructor showed the standard VM-vs-container diagram to illustrate the architectural difference.

![VM vs Container diagram](https://www.netapp.com/media/container-vs-vm-inline1_tcm19-82163.png?v=85344)

**The key structural difference:**

| Virtual Machine | Container |
|-----------------|-----------|
| Has its own full OS (kernel + everything) | Shares the host OS kernel |
| Hypervisor emulates hardware for each VM | Container runtime talks directly to the host kernel |
| Slow to start (OS boot) | Starts in milliseconds |
| Gigabytes of disk space | Megabytes — only the app and its direct dependencies |
| Completely isolated hardware-level | Process-level isolation (same kernel, different namespace) |

**Why containers can skip the OS:**
Every OS has two layers — the **kernel** (the core that talks to hardware, manages memory, processes, etc.) and the userspace software on top (shell, tools, GUI, etc.). What differs between Linux distributions is mostly the userspace — Ubuntu, Debian, Alpine, Fedora all share essentially the same Linux kernel underneath, but have different package managers, default tools, and libraries.

A container only needs the userspace parts it uses. It borrows the host machine's kernel directly. This is why a container can be a few hundred megabytes instead of 20+ gigabytes — no kernel to include.

### 3. 🐳 Docker — The Container Tool

The most widely used tool for containers today is **Docker**. It handles:
- Pulling container **images** from the internet (Docker Hub)
- Creating and running **containers** from those images
- Managing images, containers, networks, and storage

**Image vs Container — the key distinction:**

| Term | What it is |
|------|-----------|
| **Image** | A read-only blueprint/recipe — defines what the container will contain. Like an `.iso` file for a VM. Stored on disk. |
| **Container** | A running instance created *from* an image. Like an installed, running VM. Ephemeral — deleting it doesn't delete the image. |

You can create many containers from one image. When a container stops or is deleted, the image remains — the next `docker run` creates a fresh container from the same image almost instantly (no re-download needed).

### 4. 💻 Installing Docker Desktop on Windows

For Windows, the tool to install is **Docker Desktop** — a GUI application that bundles the Docker engine and a management interface.

```bash
# Install via winget
winget install Docker.DockerDesktop
```

**🔗 Links shared in class:**
| Resource | Description |
|----------|-------------|
| [Docker Desktop](https://www.docker.com/products/docker-desktop/) | Official download page |

**Two Windows features must be enabled for Docker Desktop to work:**

**1. Windows Subsystem for Linux (WSL2)**
Docker containers are Linux containers — they need a Linux kernel to run on. On Windows, Docker uses WSL2 to provide that kernel. WSL2 must be installed and enabled.

```bash
# Install WSL2 from the terminal (admin rights required)
wsl --install
```

**🔗 Links shared in class:**
| Resource | Description |
|----------|-------------|
| [Install WSL — Microsoft Learn](https://learn.microsoft.com/en-us/windows/wsl/install) | Official WSL2 installation guide |

**2. Windows Hyper-V Platform**
Hyper-V is Windows' own virtualisation feature (needed by WSL2 under the hood). It must be enabled in Windows Features. Both of these may already be on — Docker Desktop will tell you if anything is missing during setup. If it asks you to enable them, let it, and restart when prompted.

**School computers — the docker group:**
On school machines installed with a restricted account, Docker may be installed under the admin/school account, not your personal student account. To use Docker from your own account:
1. Log in with the **school admin account** (opilane or similar)
2. Open **Local Users and Groups** (`lusrmgr.msc`)
3. Go to Groups → `docker-users`
4. Double-click it → Add → search for your own account name → OK
5. Log out and log back in with your own account — the group membership takes effect on login

**Verify everything works:**
```bash
docker version     # shows Docker client and server versions
docker run hello-world   # official test — pulls and runs a tiny image
```

### 5. 🐄 First Docker Command — `docker run chuanwen/cowsay`

After Docker Desktop was confirmed working, the instructor ran the first real Docker command live:

```bash
docker run chuanwen/cowsay
```

**What happens step by step:**
1. Docker checks if the image `chuanwen/cowsay` exists locally → it doesn't (first time)
2. Docker pulls the image from **Docker Hub** — downloads all the layers
3. Docker creates a container from that image
4. The container runs its program (cowsay — a classic Linux terminal toy that prints ASCII art of a cow saying something)
5. The program finishes → the container stops automatically

The image was ~334 MB. After the first pull, running it again is instant — the image is cached locally.


**Why this is a great demo:**
No installation, no dependencies to manage, no "works on my machine" problems. One command and a program that didn't exist on your computer a second ago is running — isolated, clean, reproducible on any machine that has Docker. When the container exits, there's no cleanup needed. The image stays cached; the container itself is gone.

### 6. 🌍 Linux Distributions — A Brief Detour

The cowsay container is a Linux container. This prompted a brief sidebar on the Linux landscape — relevant because in production, 99% of servers run Linux.

**What "Linux" actually means:**
"Linux" technically refers only to the **kernel** — written by Linus Torvalds in 1991. The kernel alone is not a usable system. What people call "Linux" is actually a **distribution** (distro): the kernel combined with a set of tools, package managers, and userspace software.

There are hundreds of distros. They share the Linux kernel but differ in everything built on top — package manager, default shell, included tools, philosophy. The major families:

| Family | Key members | Notes |
|--------|-------------|-------|
| **Debian** | Debian, Ubuntu, Kubuntu, Mint, Pop!_OS | Ubuntu is the most popular for desktops and servers |
| **Red Hat** | RHEL, Fedora, CentOS, AlmaLinux | Common in enterprise/corporate servers |
| **Arch** | Arch Linux, Manjaro | Rolling release, highly customisable, famously complex |
| **Alpine** | Alpine Linux | Tiny — often used as base for Docker images |

Most Docker base images (including the cowsay one) are built on **Alpine** or **Debian/Ubuntu** because they're small and well-supported. The instructor showed Docker Hub's "Operating Systems" category — Ubuntu, Debian, Alpine, Fedora and many others all have official Docker images available.

**macOS is not Linux** — it has its own kernel (XNU/Darwin), though the command-line interface is similar enough that most Linux commands work. Many server tools that run on Linux also run on macOS, but not always identically.

### 7. 🐧 WSL2 — Linux on Windows (Bonus)

Near the end, the instructor mentioned **WSL2** (Windows Subsystem for Linux) as something worth knowing about, even beyond its use as Docker's backend.

WSL2 runs a real Linux kernel inside Windows — not a VM in the traditional sense, but a lightweight Hyper-V virtual machine that's tightly integrated with Windows: shared filesystem, shared networking, launches in seconds. You can run Ubuntu, Debian, Kali, or other distros side by side in your Windows terminal, access Windows files from Linux and vice versa, and run Linux-native tools like bash scripts and package managers.

For Docker specifically: WSL2 is *how Docker runs Linux containers on Windows*. Docker Desktop's engine runs inside a WSL2 VM. This is what makes it possible to run Linux containers on a Windows machine without running a full VM yourself.
